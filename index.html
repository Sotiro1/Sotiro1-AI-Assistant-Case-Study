Building Sotiro, a Personal AI Assistant from Scratch

A concept and architecture study for a voice-enabled, context-aware AI agent that lives on your PC. It can answer questions, access your files, listen to lectures, and take actions on your behalf.

Author Aaron Sotiropoulos
Year 2026
Domain AI / Systems / Automation
Status
CONCEPT PHASE
01

The Problem

As a software engineering student juggling university assignments, industry placements, freelance automation work, and personal projects, the mental overhead of context-switching adds up fast. Existing AI tools like Claude and ChatGPT are powerful, but they are reactive. You have to go to them. They don't know your files, your calendar, your ongoing projects, or your workflow.

This project asks a simple question: what would it look like to build an AI assistant that is proactive, persistent, and genuinely part of your daily computing environment? One that you built yourself and fully control?

"Instead of going to an AI tool, the AI should already be there. Aware of your context, accessible by voice, and able to actually do things."

🔄
Context loss between tools

Every new chat session starts from zero. The AI doesn't know what you worked on yesterday.

🖱
No computer access

Existing AI tools can't open your files, check your calendar, or run scripts on your machine.

🗣
Not voice-native

Typing is slow. A truly useful assistant should let you speak naturally and respond in kind.

🔒
No ownership

You're reliant on a third-party product. A self-built system means full control over data and behaviour.

02

The Proposed Solution

A locally-running Python application called Sotiro that combines a large language model (Claude API) with a voice interface, a persistent memory store, and a set of tools that let it interact with the operating system and external services.

The assistant is built around four core capabilities. Conversation: ask it anything, and it remembers the context. Voice: speak to it and hear it respond. Memory: it knows your projects, schedule, and preferences. Action: it can open files, run scripts, search the web, listen to lectures, and trigger automations via n8n.

Rather than build everything at once, the project is broken into five incremental phases, each independently useful and each building on the last. The first phase is deliberately minimal: a Python script that talks to the Claude API with a custom system prompt. Everything else grows from there.

03

Architecture

The system is structured as four layers. Data flows upward from input to the LLM brain, and downward from the LLM through tools to produce real-world outputs.

Input
Voice / Text
Whisper STT or keyboard
Brain
LLM — Claude API
System prompt · conversation history
Memory
Memory Store
ChromaDB · knowledge file
Actions
Files
Read / write OS
Browser
Playwright
Scripts
subprocess
n8n
Automations via webhook
Output
Voice / Text Response
ElevenLabs TTS or Piper

The orchestration layer is a custom Python agent loop that sits between the LLM and the action layer. It parses tool calls from the LLM response, executes the right function, and feeds the result back into the conversation context. This is the same pattern used by Claude's tool use API and by frameworks like LangChain.

04

Technology Stack

The stack prioritises tools that are free or low-cost, run locally where possible, and integrate well with Python. This reflects the reality of building this as a side project alongside a full-time placement.

Layer Tool Rationale
Language Python 3.12 Dominant language in the AI/ML ecosystem. All major libraries available.
LLM Claude API Strong tool-use support, long context window, existing familiarity from Nisteia project.
Speech-to-text OpenAI Whisper Free, runs locally, high accuracy across accents. No data sent to the cloud.
Text-to-speech ElevenLabs or Piper ElevenLabs for voice quality with a free tier. Piper as a fully local fallback with no internet required.
Meeting listener Whisper + pynput Whisper handles transcription. pynput listens for the hotkey to start and stop recording without any visible UI.
Memory / RAG ChromaDB Lightweight local vector database. Simple Python API. No server required.
GUI automation pyautogui Cross-platform mouse/keyboard control for desktop app interaction.
Browser automation Playwright Reliable headless browser control. Existing familiarity from testing coursework.
Workflow automation n8n Existing workflows from LDC Strategies placement can be reused via webhook triggers.
Orchestration Custom agent loop Lightweight Python loop to handle tool dispatch and conversation history. No framework overhead.
05

Build Roadmap

The project is broken into six self-contained phases. Each one ships something usable on its own, so there is no point where Sotiro is half-built and not worth running. The TAE Aerospace placement runs in parallel, so phases are intentionally scoped for evenings and weekends.

01
Text-only Claude agent

Python script with Claude API, a personal system prompt, and two tools: read a file, fetch the weather. Proves the core loop works before adding anything else.

⬤ STARTING POINT
02
Voice interface

Add Whisper for microphone input and ElevenLabs (or Piper) for spoken responses. Sotiro can now be spoken to naturally, without typing.

◯ PHASE 2
03
Wake-on-login and daily briefing

Sotiro launches automatically when the laptop opens and greets you by name with a spoken summary of your day: calendar events, unread emails worth knowing, and any deadlines coming up. No prompt needed.

◯ PHASE 3
04
Lecture and meeting listener

A single hotkey starts Sotiro recording in the background. It transcribes the audio using Whisper and, when you stop it, produces a clean summary with key points and any dates or action items mentioned. A companion phone shortcut handles the same when the laptop is closed.

◯ PHASE 4
05
Persistent memory

ChromaDB stores past conversations and a knowledge file holds personal context: current projects, preferences, and schedule. Sotiro now remembers across sessions and can answer questions about what you did last week.

◯ PHASE 5
06
Action layer and n8n bridge

Expose tools for file I/O, running scripts, and browser control via Playwright. Then connect to existing n8n workflows via webhooks so Sotiro can send emails, update records, post to Slack, and trigger multi-step automations on command.

◯ PHASE 6
06

Anticipated Learnings

As a concept-stage project, the lessons below are forward-looking. They reflect the hard problems expected before a line of code is written, and will be updated as each phase is completed.

01
Agent loop design is non-trivial

Managing conversation history, tool call parsing, and context window limits in a production-quality loop requires careful design from the start.

02
Voice latency is a UX problem

The round-trip from speech → STT → LLM → TTS → speech introduces latency. Making this feel natural will require streaming and careful prompt design.

03
System prompt engineering matters

The quality of the assistant is heavily determined by the system prompt. Crafting one that gives the model personality, context, and reliable behaviour is its own discipline.

04
Safety and access control

An assistant that can run scripts and access files is a significant security surface. Building in confirmation steps and scoped permissions from phase 4 onwards is essential.
